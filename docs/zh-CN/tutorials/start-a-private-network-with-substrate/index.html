Start a Private Network with Substrate

在这个教程中，我们将学习和实践如何使用Substrate启动一个区块链网络，并使用你选择的验证者/管理者集合。

你需要：

  1. 一个 Linux 或 Mac 操作系统下的命令行客户端
  2. Git

您将学习到：

  1. 从源代码编译一个 Substrate 节点。
  2. 为网络验证人生成 ed25519 密钥对
  3. 创建和编辑 chainspec json 文件

编译所需工具

Substrate 是一个开源框架，用来构建你自己的区块链。 可以用来开发各种各样的应用程序。

Substrate (目前) 没有提供二进制安装包，因此必须从源代码中编译它，这可能是一个耗时的过程。 Substrate 仓库 会经常更新，请确保参加此网络的每个人都有同样版本的 Substrate 以保证成功。 在实践中，类似但不完全相同的版本通常可以正常工作，但是依靠这个特点通常会让人感到沮丧。 我们将使用 v1.0 分支来获取更多的稳定性。

首次操作

首先，我们将通过克隆 git 仓库下载 Substrate v1.0 代码。

git clone -b v1.0 https://github.com/paritytech/substrate
cd substrate

仓库包含了一个名为 subkey 的工具，我们可以选择用它来生成和检查密钥对。 让我们先编译它。 --force 选项意味着我们将移除之前安装的版本，重新安装此版本。

cargo install --force --path subkey subkey

现在让我们编译我们将要运行的区块链节点。 由于 Substrate 是一个框架，大多数现实世界中的区块链将使用它编写自定义的 runtime 代码。 有 其他教程 详细涵盖这方面的内容。 幸运的是，Substrate 仓库本身已经有两个随时可以运行的节点环境。 第一个在 node 目录，它包含了许多功能，可以构建实际可行的区块链。 事实上，它看起来和 Polkadot 类似，Polkadot 也是基于 Substrate 构建的。 第二个是在 node-template 目录下的最小化 runtime 代码。 我们将在本教程中使用节点模板 node-template ，因为它比较简单，通常是编写自定义 runtime 的起点。

# Switch to node-template directory
cd node-template

# Ensure your Rust toolchain is up to date
./scripts/init.sh

# Compile the Wasm version of the runtime code
./scripts/build.sh

# Compile the native version of the node
cargo build --release

下次更新

更新或改变 Substrate 版本时，这个过程类似，但构建速度更快。 首先，获取最新代码，即使你最终会使用某个旧版本。

# 确保在 v1.0 分支
git checkout v1.0

# 获取最新代码
git pull

Subkey doesn't change nearly as quickly as Substrate itself, so you can often skip rebuilding it. 如果你收到 checksum 错误或密钥兼容性错误，请确保 subkey 是最新的。

# 可选: 重新编译 subkey
cargo install --force --path subkey subkey

现在我们准备重新编译节点模板。 这个过程与以前相同，但运行速度要快，因为只有实际改变的代码部分会被重新编译。

cd node-template
./scripts/init.sh
./scripts/build.sh
cargo build --release

提示：如果想要 node-template 安装在可执行文件路径下，你可以在上一步使用 cargo install 代替 cargo build。

Alice 和 Bob 启动区块链

在我们生成自己的密钥并开始一个真正独特的 Substrate 网络之前，让我们了解下基础知识，首先使用一个预先定义的网络规范，称为 local ，使用两个预先定义的密钥 (当然不是私人的!)，称为 Alice 和 Bob。

Alice 首先启动

Alice 应该从 Substrate 仓库的根目录运行此命令。

# 如果你仍然在 node-template/ 目录
cd ..

# Start the node
./target/release/node-template \
  --base-path /tmp/alice \
  --chain=local \
  --key //Alice \
  --port 30333 \
  --telemetry-url ws://telemetry.polkadot.io:1024 \
  --validator \
  --name AlicesNode

Let's look at those flags in detail:

  • --base-path 指定一个目录，Substrate 用它来存储与此链有关的所有数据。 如果目录不存在，将创建它。 如果已经存在其它区块链的数据，启动将会报错。 这种情况下，要么清除目录，要么选择不同的目录。
  • --chain=local 指定要使用的链规范。 有几个预先准备的选项，包括 local development 和 staging ，但更常见的是我们指定自己的 chainspec 文件。 我们将在稍后阶段指定自己的文件。
  • --key //Alice 指定我们正在使用预定义的 Alice 密钥作为此节点的验证者。 Generally one should generate their own key with subkey and then the flag would look like --key "enroll mechanic science ..."". 我们将在稍后阶段指定自己的秘钥。
  • --port 30333 指定节点将监听的 p2p 端口。 30333 is the default and this flag can be omitted if you're happy with the default. 如果 Bob 的节点在同一物理机上运行，您需要明确指定一个不同的端口。
  • --telemetry-url 告诉节点，将监测数据发送到特定的服务器。 我们在这里选择了由 Parity 维护、可供任何人使用的监测服务器。 You may host your own (beyond the scope of this article) or omit this flag entirely.
  • --validator 表示我们的节点将参与生产区块而不是仅仅同步网络数据。
  • --name 给节点添加一个可读性高的名字。 This flag is optional and if you omit it a human-readable name will be generated for you. 如果你不使用监测服务器，你会发现为节点指定名字没有多少价值。

连接UI

你可以通过节点在命令行中产生的输出了解很多信息。 还有一个很好的图形用户界面，称为 Polkadot Js Apps UI ，可以将它连接到你的节点。 此链接会跳转到该UI的官方托管网页，非常方便，但与你正在运行的 Substrate 版本相比可能已经过时。 在托管的UI页面过期的情况下，你可以通过从 github 获取代码在本地运行该应用。 一般来说，该仓库中的指令将是你的最佳指南，但过程应该是这样的。

# 抓取代码
git clone https://github.com/polkadot-js/apps
cd apps

# 安装依赖
yarn

# 启动项目
yarn run start

在你常用的Web浏览器输入 localhost:3000，然后在设置页面上选择“本地节点”。 你应该看到类似下面的内容：

你会注意到，在命令行和用户界面中，还没有生产任何块。 好奇的学习者请参考 这里。 一旦另一个验证者加入网络，块将开始生成。

Bob 加入

现在，Alice节点已经启动并运行，Bob 可以通过引导节点来加入网络。 他的命令看起来非常类似下面：

./target/release/node-template \
  --base-path /tmp/bob \
  --chain=local \
  --key //Bob \
  --port 30333 \
  --telemetry-url ws://telemetry.polkadot.io:1024 \
  --validator \
  --name BobsNode \
  --bootnodes /ip4/<Alices IP Address>/tcp/<Alices Port>/p2p/<Alices Node ID>

上面已经解释了大多数这些选项，但有几个要点。

  • 如果这两个节点在同一物理机器上运行，Bob 必须指定了一个不同的 --base-path 和 --port。
  • Bob has added the --bootnodes flag and specified a single boot node, namely Alice's. 他必须正确指定这三项，Alice 可以为他提供的这三条信息。
    • Alice IP Address 的格式 192.168.1.1
    • Alice的端口，可能为30333
    • Alice节点ID，从她的日志输出中复制。 (QmNTx7qyy5yd5JisUTzjaAbTCUeH5ot9xMjRejRejPRjljlSED7MY 在上面的输出。)

如果所有进展顺利，在几秒之后，节点应该相互连接，开始生成区块。 你应该看到类似下面的内容：

Idle (1 peers), best: #1 (0x9f1b…9b57), finalized #0 (0x28be…45e5), ⬇ 1.7kiB/s ⬆ 1.4kiB/s

此行显示，Bob 有对等节点 Alice (1 peers)，它们生产了一个模块(best: #1 (0x9f1b…9b57))，并且区块尚未最后确定/敲定 (finalized #0 (0x28be…45e5))。 在这个网络中，只有创世区块被最终敲定，因为节点模板中没有最终性模块（finality gadget）。 最终性（Finality）超出本教程的范围。

生成密钥

现在我们知道了基本原理和命令行选项，现在是时候生成我们自己的密钥而不是使用众所周知的 Alice 和 Bob 密钥。

每个想要参与区块链的人都可以使用我们之前使用的subkey工具或Polkadot JS Apps UI生成自己的密钥。 在这个教程中，我们将使用 UI。

导航到 "Accounts"（账户）选项卡，并选择 "Add Account"（添加帐户）。 对于这种会话密钥（session key），我们需要选择底部附近的 Edwards Curve，并复制粘贴公钥地址和原始种子。

当您生成了一个密钥对，将整个输出存储在安全位置。 与网络中的其他验证人共享你的公钥地址。 当网络中的每个人都知道其他人的地址时，你就可以发布自己的链了。

启动你的私有区块链

上一次，我们使用--chain=local这是一个预定义的“chainspec”，它将 Alice 和 Bob 指定为验证人以及许多其他有用的默认值。

创建 chainspec

我们不是从头开始完全编写 chainpec，而是对我们之前使用的那些进行一些修改。 首先，我们需要将 chainspec 导出到 json 文件。 请记住，运行node-template --help可以获得有关所有这些命令的更多详细信息。

./target/release/node-template build-spec --chain=local > customSpec.json

我们刚创建的文件包含几个字段，人们可以通过探索它们来学习很多东西。 By far the largest field is a single hex number that is the Wasm binary of our runtime. 这是你在运行build.sh脚本时构建的。 从 Gav的web3峰会演示详细了解那些信息上链有用的原因。

The portion of the file we're interested in is the authorities which looks like this

  "consensus": {
      "authorities": [
        "5FA9nQDVg267DEd8m1ZypXLBnvN7SFxYwV7ndqSYGiN9TTpu",
        "5GoNkf6WdbxCFnPdAnYYQyCjAKPJgLNxXwPjwTh6DGg6gN3E",
      ],
      "code": <snip>
    }

All we need to do is change the authority addresses listed (currently Alice and Bob) to our own addresses that we generated in the previous step. 一个人应该执行这些步骤并共享生成的文件。 与其他验证人一起。 (Because Rust -> Wasm builds aren't "reproducible", each person will get a slightly different Wasm blob which will break consensus if each participant generates the file themselves.)

准备好 chainspec 后，将其转换为“raw” chainspec。 regular 和 raw 之间的区别只是所有字段名称在 raw chainspec 中编码为十六进制。 在那里探索没什么好玩的。

./target/release/node-template build-spec --chain customSpec.json --raw > customSpecRaw.json

最后与网络中的所有其他验证人共享customSpecRaw.json。

启动链

你已完成所有必要的准备工作，现在准备好启动链。 此过程非常类似于之前以 Alice 和 Bob 的身份启动链。 从干净的基础路径开始非常重要，如果你打算使用之前的路径，请删除该目录中的所有内容。

第一位参与者启动节点

./target/release/node-template \
  --chain ./customSpecRaw.json \
  --key "enroll mechanic ... embody" \
  --port 30333 \
  --telemetry-url ws://telemetry.polkadot.io:1024 \
  --validator

以下是我们启动 Alice 时的一些不同之处。

  • I've omitted the --base-path flag, so Substrate will use a default location. 你仍然可以自由地指定路径。
  • 对于密钥，我使用了之前生成的助记词。 这必须是与你的 chainspec 中的某个地址对应的助记词。
  • The --chain flag has changed to use our custom chainspec.

Subsequent validators can now join the network as Bob did previously, making sure to use the new chainspec and key. 你可以从任意网络中已有的节点引导，而不仅仅是先前的节点。

All validators must be using identical chain specifications in order to peer together.

后续步骤

恭喜你！ 你已启动了自己的区块链！

在这个教程中，你学会了编译节点模板，生成自己的公私钥对，创建一个自定义的链，使用这些密钥对，并根据自定义 chainspec 和节点模板启动了一个私有区块链网络。

以下是接下来要学习的内容的一些想法。

使用全节点

你可以使用位于node目录中的 Substrate 的全节点，而不是我们使用的node-template，完成一个非常类似的过程。 你应该熟悉该过程，并且将开始探索全节点中包含的更多 runtime 模块。

创世后添加验证人

如果想在启动区块链后添加更多验证人，该怎么办？ 你不能再返回并编辑那里的 chainspec。 The authority set can be changed after genesis by calling the consensus module's set_authorities function. 但是，该函数只能从其它 runtime 模块调用。 如果你已构建全节点，则可以使用 democracy 模块完成该任务。 或者查看Substrate POA runtime。

编写一个自定义 Runtime

Substrate 其实就是写自己的 runtime，现在你已经上手了，我鼓励你尝试写一个。 Consider one of the tutorials.
