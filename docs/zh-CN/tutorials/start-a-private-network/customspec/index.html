创建自定义的 chain spec

当所有参与者都生成了自己的密钥后，你就可以开始创建自定义的 chain spec。 我们将使用这个自定义 chain spec 来代替之前使用的内置local spec。

本例中，我们将创建一个双节点网络，但这个过程可以直接泛化到更多节点。

创建一个 Chain Spec

上一次，我们使用了 --chain local 参数，它是一个预定义的 chain spec，里面将 Alice 和 Bob 指定为验证节点及有其他默认值。

我们无需从头开始写新的 chain spec，可以对之前使用的 chain spec 进行一些修改。 首先，我们需要将 chain spec 导出到一个名为 customSpec.json 的文件中。 请记住，关于所有这些命令的更多细节可以通过运行 node-template --help 获得。

# Export the local chain spec to json
./target/release/node-template build-spec --disable-default-bootnode --chain local > customSpec.json

我们刚刚创建的文件中包含了几个字段，通过对它们的探索，你可以学到很多东西。 到目前为止，最长的字段是一个单一的二进制代码块，就是我们 runtime 的 Wasm 二进制文件。 它是之前运行 cargo build --release 命令时，构建的内容的一部分。

我们感兴趣的这部分文件内容，由下面的 "aura" 字段显示，是用于创建区块的 Aura 权限，以及由 "grandpa" 字段显示，用于最终确定区块的 GRANDPA 权限。 That section looks like this using the two provided demo keys:

{
 //-- snip --
 "genesis": {
    "runtime": {
      "frameSystem": {
        //-- snip --
      },
      "palletAura": {
        "authorities": [
          "5FfBQ3kwXrbdyoqLPvcXRp7ikWydXawpNs2Ceu3WwFdhZ8W4",
          "5EhrCtDaQRYjVbLi7BafbGpFqcMhjZJdu8eW8gy6VRXh6HDp"
        ]
      },
      "palletGrandpa": {
        "authorities": [
          ["5G9NWJ5P9uk7am24yCKeLZJqXWW6hjuMyRJDmw4ofqxG8Js2", 1],
          ["5CRZoFgJs4zLzCCAGoCUUs2MRmuD5BKAh17pWtb62LMoCi9h", 1]
        ]
      },
    //-- snip --
    }
 }
}

这部分说明是为 node template 而写的。 它们只需进行少量修改，就能与其他基于 Substrate 开发的节点一起交互。 In Substrate, nodes that include the session pallet should leave the Aura and Grandpa configs empty and instead insert this information in the session config. All of this is demonstrated in the chain spec you export from your desired node.

由于 grandpa 协议支持加权投票，所以 grandpa 的数据格式更为复杂。 在这种情况下，我们给每个验证节点的权重为 1。

我们需要做的就是将列出的权限地址 (目前是 Alice 和 Bob) 改为我们自己在上一步中生成的地址。 sr25519 地址放在 aura 部分，ed25519 地址放在 grandpa 部分。 你可以根据自己的需求添加任意数量的验证节点。 想了解更多，可查看 Substrate 里的密鈅。

即使是抱着学习的目的，验证节点也不应该共享相同的密钥。 如果两个验证节点具有相同的密钥，它们产生的区块将互相冲突。

当 chain spec 准备好后，我们将其转换为 "原生" chain spec。 原生 chain spec 包含着所有相同的信息，但它同时也包含节点用于引用本地存储数据的已编码的存储密钥。 部署原生 chain spec 可以确保每个节点用适当的存储密钥对数据进行存储。

./target/release/node-template build-spec --chain=customSpec.json --raw --disable-default-bootnode > customSpecRaw.json

最后，将 customSpecRaw.json 与网络中的所有其他验证节点共享。

A critical note: A single person should create the chain spec and share the resulting customSpecRaw.json file with their fellow validators.

Because Rust -> Wasm optimized builds aren't "reproducible", each person will get a slightly different Wasm blob which will break consensus if each participant generates the file themselves. For the curious, learn more about this issue in this blog post.
