添加节点授权模块

关于 node-authorization pallet

The node-authorization pallet is a build-in pallet in Substrate FRAME, which manages a configurable set of nodes for a permissioned network. 每个节点通过 PeerId 进行标识，它仅仅包含了 Vec<u8> 这一数据类型。 Each PeerId is owned by an AccountId that claims it (these are associated in a map). 通过这个模块，你有两种方法来授权节点加入到区块链网络中：

  1. 加入到 well known 节点集合中，这些节点彼此可以连接。 这一方式通常要经过系统中治理机制的同意。
  2. 请求允许连接到某个已经加入到网络中的节点。 这一节点可以是 well known 节点，也可以是普通节点。

通过 PeerId 关联的某个节点 有且仅有一个拥有者。 Well known 节点的拥有者是在添加它们时指定的。 如果是普通节点，任何 用户都可以 claim 认领这一 PeerId 作为对应节点的拥有者。 为了防止被其他人认领，节点的维护人员应当在启动该节点之前就去认领（当然，需要你已经拿到了对应的 PeerId）。

之后，拥有者可以为自己的节点修改额外的连接。 需要指出的是，拥有者不能修改 well known 节点之间的连接，因为它们之间始终是允许连接的。 Instead, you can manipulate the connection between a well know node and a normal node or between two normal nodes.

在 node-authorization pallet 中使用了 链下工作机来设置授权的节点。 请使用相应的命令行参数确保链下工作机处于开启状态，这是因为对于非验证人节点链下工作机默认是关闭的。 你的节点有可能无法同步到最新区块，如果出现这种情况，关闭链下工作机并且手动指定允许连接的节点来完成网络同步。

添加 node-authorization pallet

If you already hace Node Template cloned, you can just create a new branch and check it out from the base template, otherwise, clone v3.0.0 of the project:

# Fresh clone, if needed:
git clone -b v3.0.0 --depth 1 https://github.com/substrate-developer-hub/substrate-node-template
# From the working directory, create a new branch and check it out
cd substrate-node-template
git branch perm-network
git checkout perm-network

你应该可以 check (或者 build) 成功：

cd substrate-node-template/
make init
make check

如果在编译时遇到了问题，参考这些帮助信息

现在可以用你最喜欢的编译器打开代码，等不及了是不是？

在 runtime/Cargo.toml 中，添加我们模块的依赖：

runtime/Cargo.toml

[dependencies]
#--snip--
pallet-node-authorization = { default-features = false, version = '2.0.0' }

#--snip--
[features]
default = ['std']
std = [
    #--snip--
    pallet-node-authorization/std,
    #--snip--
]

现在我们在 runtime/src/lib.rs 中引入和使用我们的模块。 首先引入 use frame_system::EnsureRoot，在本文这个简单的区块链中，我们需要用它来模拟治理机制。 之后实现模块的配置接口。 More description on the trait can be found in its reference doc.

runtime/src/lib.rs


/* --snip-- */

use frame_system::EnsureRoot

/* --snip-- */

parameter_types! {
    pub const MaxWellKnownNodes: u32 = 8;
    pub const MaxPeerIdLength: u32 = 128;
}

impl pallet_node_authorization::Trait for Runtime {
    type Event = Event;
    type MaxWellKnownNodes = MaxWellKnownNodes;
    type MaxPeerIdLength = MaxPeerIdLength;
    type AddOrigin = EnsureRoot<AccountId>;
    type RemoveOrigin = EnsureRoot<AccountId>;
    type SwapOrigin = EnsureRoot<AccountId>;
    type ResetOrigin = EnsureRoot<AccountId>;
    type WeightInfo = ();
}

/* --snip-- */

最后，就可以把我们的 pallet 放在 construct_runtime 宏里，使用下面一行代码：

runtime/src/lib.rs

construct_runtime!(
    pub enum Runtime where
        Block = Block,
        NodeBlock = opaque::Block,
        UncheckedExtrinsic = UncheckedExtrinsic
    {
        /* --snip-- */

        /*** Add This Line ***/
        NodeAuthorization: pallet_node_authorization::{Module, Call, Storage, Event<T>, Config<T>},

        /* --snip-- */

    }
);

为我们的模块添加创世存储配置

PeerId is encoded in bs58 format, so we need a new library bs58 in node/cargo.toml to decode it to get its bytes.

node/cargo.toml

[dependencies]
#--snip--
bs58 = "0.3.1"
#--snip--

现在我们在 node/src/chain_spec.rs 中添加合适的创世存储。 类似地引入必要的依赖：

node/src/chain_spec.rs

/* --snip-- */
use sp_core::OpaquePeerId; // A struct wraps Vec<u8>, represents as our `PeerId`.
use node_template_runtime::NodeAuthorizationConfig; // The genesis config that serves for our pallet.
/* --snip-- */

在帮助函数 testnet_genesis 中添加我们的创世配置，

/// Configure initial storage state for FRAME modules.
fn testnet_genesis(
    wasm_binary: &[u8],
    initial_authorities: Vec<(AuraId, GrandpaId)>,
    root_key: AccountId,
    endowed_accounts: Vec<AccountId>,
    _enable_println: bool,
) -> GenesisConfig {

    /* --snip-- */

    /*** Add This Block Item ***/
    pallet_node_authorization: Some(NodeAuthorizationConfig {
        nodes: vec![
            (
                OpaquePeerId(bs58::decode("12D3KooWBmAwcd4PJNJvfV89HwE48nwkRmAgo8Vy3uQEyNNHBox2").into_vec().unwrap()),
                endowed_accounts[0].clone()
            ),
            (
                OpaquePeerId(bs58::decode("12D3KooWQYV9dGMFoRzNStwpXztXaBUjtPqi6aU76ZgUriHhKust").into_vec().unwrap()),
                endowed_accounts[1].clone()
            ),
        ],
    }),

    /* --snip-- */

}

NodeAuthorizationConfig 包含了一个属性叫做 nodes，类型是元组的集合。 元组的第一个元素是 OpaquePeerId，我们使用了 bs58::decode 把可读性友好的 PeerId 格式转换成字节集合。 元组的第二个元素是 AccountId，代表了节点的拥有者，这里我们使用的是预置的账户。 也就是说，第一个节点的拥有者是 Alice，而 Bob 拥有第二个节点。

你可能好奇 12D3KooWBmAwcd4PJNJvfV89HwE48nwkRmAgo8Vy3uQEyNNHBox2 是从哪来的。 我们可以使用 subkey 工具来生成这个可读性强的 PeerId。

subkey generate-node-key

说明：subkey 是 Substrate 提供的一个命令行工具，也可以在本机直接安装！ - Install Intructions

这一命令的输出为：

12D3KooWBmAwcd4PJNJvfV89HwE48nwkRmAgo8Vy3uQEyNNHBox2 // this is PeerId.
c12b6d18942f5ee8528c8e2baf4e147b5c5c18710926ea492d09cbd9f6c9f82a // This is node-key.

现在，所有的代码修改都已经完成了，可以去启动我们的许可网络了。 休息一下再继续吧！
